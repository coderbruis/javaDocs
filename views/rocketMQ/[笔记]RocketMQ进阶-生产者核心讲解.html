# 1. RocketMQ生产者核心参数详解

producerGroup：生产者组名

对于生产者来说，group是唯一存在的，如果启动两个相同group的生产者，会报错。

createTopicKey

一般不轻易创建topic（不会给开发者生产topic，需要通过工单创建topic）

defaultTopicQueueNums：默认队列个数（四个）

sendMsgTimeout（单位：ms）：消息发送超时时间

compressMsgBodyOverHowmuch (默认压缩字节4096)

retryTimesWhenSendFailed (可配置)：重发配置

retryAnotherBrokerWhenNotStoreOK (默认false)：未存储成功则尝试其他的broker进行消息存储

maxMessageSize (默认128K)：最大消息大小

# 2. 主从同步机制分析

Master - Slave主从同步

RocketMQ中主从同步机制主要同步哪些内容呢？

在RocketMQ中主要同步两种内容：数据内容 + 元数据信息

元数据信息包括：Topic信息、消费端消费进度等

而数据内容包括了CommitLog等内容

元数据同步：会对Broker角色进行识别，如果是Slave则启动同步定时任务

数据内容同步：CommitLog数据的同步是实时的（CommitLog是全量的，重要的数据）

消息同步核心类：HAService、HAConnection、WaitNotifyObject

元数据同步底层使用的Netty实现的，数据内容的同步是使用的原生socket实现的。

# 3.1 元数据同步流程

  1. BrokerStartup#main
  2. BrokerStartup#start
  3. handleSlaveSynchronize
  4. 启动定时任务，BokerController.this.slaveSynchronize.syncAll
  5. SlaveSynchronize#syncAll()(同步topic配置，同步消费者偏移量，同步延迟偏移量，同步订阅组配置信息)
  6. SlaveSynchronize#syncTopicConfig
  7. BrokerOuterAPI#getAllTopicConfig
  8. NettyRemotingClient#invokeSync
  9. NettyRemotingAbstract#invokeSyncImpl（此时进行Netty的channel#writeAndFlush调用）

# 3.2 消息内容同步

  • Master节点

    • AcceptSocketService：接受slave节点连接
    • HAConnection
      • ReadSocketService：读来自slave节点的数据
      • WriteSocketSErvice：写到slave节点的数据

  • Slave节点

    • HAService
      • HAClient：对Master节点连接、读写数据

通信协议：Master节点与Slave几点通信协议很简单，只有两条：

对象 用途 第几位 字段 数据类型 节点 说明
Slave => Master 上报CommitLog已经同步到的物理位置 0 maxPhyOffset Long 8 CommitLog最大物理位置
Master => Slave 传输新的CommitLog数据 0 fromPhyOffset Long 8 CommitLog开始物理位置
1 size Int 4 传输CommitLog数据长度
2 body Bytes size 传输CommitLog数据

数据消息同步流程

  1. BrokerStartup#main
  2. BrokerStartup#start
  3. DefaultMessageStore#start
  4. HAService#start
  5. HAClient#start
  6. 调用HAClient#run方法
  7. HAService#connectMaster
  8. RemotingUtil#connect （进行NIO底层调用）
  9. this.selector.select(1000)监听读写事件
  10. HAService#processReadEvent
  11. HAService#dispatchReadRequest

数据消息同步核心逻辑是判断Master的CommitLog和Slave的CommitLog偏移量offset长度是否一致，
